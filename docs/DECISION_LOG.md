### Engineering Milestone 12: Cart (`06_CART_SPECIFICATION.md`) — fulfillment-leg grouping, pricing transparency, and a genuine Medusa Cart-endpoint limitation worked around

- **Decision:** Built Cart next, per the confirmed specification implementation order. Implemented §5/§6's own "most consequential" requirement — a mixed cart renders as two visually distinct, separately-subtotaled fulfillment-leg groups (Wine & Spirits / Food Central), each with its own delivery-scope messaging, while remaining one cart and one checkout action — plus §7's quantity control (a shared component, not a second pattern), §12/§23's availability/removal labeling, and the Pricing Transparency table's "never ₦0, blank, or silently omitted" rule for shipping/tax before checkout supplies a real address.
- **Reasoning:** The vendored starter cart used a `<select>`-based quantity dropdown (already superseded by Milestone 11's PDP stepper, so extracting a shared `QuantityStepper` component satisfied §7's explicit "the cart does not introduce a second quantity-control pattern" instruction directly) and a `CartTotals` component that rendered literal "Shipping: ₦0.00" / "Taxes: ₦0.00" before checkout ever ran — a real violation of Pricing Transparency, since that reads as a false free-shipping/no-tax claim rather than "not yet known." Both were fixed as part of implementing the specification, not as unrelated cleanup.
- **Date:** 2026-07-23
- **Impact:**
  1. **Fulfillment-leg grouping** (`storefront/src/modules/cart/templates/items.tsx`): line items are split by the same `food_details`/`wine_details` presence check used throughout the platform (Product Listing card, Search badge, PDP), each group getting its own heading, delivery-scope copy (mirroring the PDP's `TrustDeliveryInfo` wording for consistency), and subtotal. Presentation only — still a single cart and a single checkout action, never order-splitting.
  2. **A genuine Medusa v2 API limitation was discovered and worked around, not silently patched over.** The Cart endpoint's own `items.variant.inventory_quantity` field expansion (`fields: "+items.variant.inventory_quantity"`) does not resolve — confirmed via direct `curl` inspection bypassing all Next.js caching, returning `null` for a variant with genuine, real stock — even though the identical `+variants.inventory_quantity` syntax resolves correctly and reliably on the Store Product endpoint (`/store/products`), which every other part of this codebase already depends on. Rather than continuing to debug Medusa's internal query-graph behavior, the cart's server component now performs a supplementary `listProducts` call for the product IDs present in the cart (the proven-reliable path) and passes real per-variant stock down to each line item as a prop — the cart's own `variant.inventory_quantity` field is no longer trusted for availability or quantity-cap decisions. Verified end to end: a genuinely depleted variant correctly shows "Currently unavailable" and a genuinely in-stock variant does not, confirmed via both a real browser test and direct API cross-checks.
  3. **Two more nested-interactive-element bugs found via real axe-core auditing of the cart page**, the same defect class first found and fixed in Milestone 6's cart-dropdown trigger: `Summary`'s "Go to checkout" and `SignInPrompt`'s "Sign in" both wrapped a vendored `Button` inside a `LocalizedClientLink`, producing a link containing a second, redundant interactive element. Fixed both by applying the button's own visual classes directly to the link instead of nesting.
  4. **A third, narrower accessibility gap found and fixed**: a cart line item's thumbnail-only link had no accessible name whenever the underlying product has no image — the shared `Thumbnail` component's fallback placeholder icon carries no alt text or label of its own. Fixed within this milestone's own file (`cart/components/item/index.tsx`) by adding a real `aria-label` (the item's own name) to the link. The same underlying gap likely exists in other thumbnail-only links elsewhere in the codebase (order history, mini-cart) — not fixed here, since those belong to already-shipped milestones and touching them would be scope creep; noted for whoever next touches those files.
  5. **Two pre-existing, site-wide color-contrast shortfalls were found and deliberately left unaltered**: the shared `Button` component's primary variant (`bg-primary`/`text-surface-elevated`, ≈4.24:1, already documented as systemic in Milestones 7/8) now also confirmed on the cart's own checkout button; and a vendored discount-code control's `text-ui-fg-interactive`. Both trace to established design tokens used across the entire site, not introduced by this milestone — changing either is a Design-System-level decision, not Cart's to make unilaterally.
  6. **`DeleteButton` gained a `data-testid` prop** for consistency with every other interactive element in the cart (matching the existing `product-quantity-value`/`product-row`/`product-title` convention) — a small, low-risk addition surfaced while writing this milestone's own structural accessibility checks, not a functional change.
  7. **Temporary QA seed data, created and fully removed within this milestone, using the now-established clean cross-deletion pattern.** Three products (an in-stock wine, a wine deliberately depleted to zero stock *after* being added to cart — simulating §12's "an item becomes unavailable" scenario for real, not by construction — and a Food Central dish) plus this milestone's first-ever cart-context stock location were created via a `claude-qa5@liquorcentral.ng` admin account, all deleted after screenshots; a second throwaway `claude-qa6@liquorcentral.ng` account was created solely to cross-delete `claude-qa5` (and was itself then deleted using `claude-qa5`'s still-live token), leaving zero leftover QA admin accounts, zero leftover QA products, and zero leftover stock locations. Milestone 9's original `claude-qa@liquorcentral.ng` leftover remains the one still-open manual cleanup item from before this pattern was adopted, unaffected by this milestone.
  Real axe-core testing against the empty and mixed-populated cart states found and fixed all violations resolvable within this milestone's scope (nested-interactive links, the thumbnail link-name gap) down to only the two pre-existing site-wide contrast issues named above. Structural checks confirmed: zero nested interactive elements, a working `role="status" aria-live="polite"` cart-total live region, three correctly-configured `role="spinbutton"` quantity controls (including a genuine `aria-valuemax` matching real per-item stock), specific per-item delete-button accessible names (`"Remove [product name]"`, never a bare "Remove item"), and full keyboard operability of the checkout link and quantity controls.
  `docs/PROJECT_STATUS.md` (→ v5.8), `docs/AI_HANDOFF.md` (→ v5.4), `docs/ROADMAP.md` (→ v6.0), and `docs/CHANGELOG.md` (new entry) updated to reflect completion; `storefront/README.md` gained a full "Milestone 12" section.
- **Status:** Final for Milestone 12. No Frozen specification was edited. Saved-for-Later (§14), Gift Wrapping (§15), and cross-selling/pairing suggestions (§18) were all deliberately not built, each blocked on an unbuilt backend mechanism the specification's own text already names as not yet built — named as real, tracked gaps rather than silently skipped. The next unblocked step, per the confirmed specification order, is Checkout (`07_CHECKOUT_SPECIFICATION.md`), now genuinely unblocked since Paystack is the approved payment provider.

### Engineering Milestone 10: Search (`03_SEARCH_SPECIFICATION.md`) — native unified-list-with-catalog-badge implementation, Meilisearch mechanisms still deferred

- **Decision:** Built Search next, per the specification implementation order Paul confirmed alongside the prior business-decision batch (Navigation → Homepage → Search → Product Listing → ...) — even though Product Listing (Milestone 9) had already been completed slightly ahead of Search in that nominal sequence. Implemented everything in `03_SEARCH_SPECIFICATION.md` achievable without Meilisearch, centered on the batch's own new requirement: a unified result list across both catalogs with a small catalog-identity badge per result, never split into separate result pages.
- **Reasoning:** Meilisearch remains unapproved (confirmed explicitly in the prior batch-of-decisions entry, above) — nothing in this milestone changes that. But re-reading §5–§25 against that constraint found a genuine, buildable subset: a unified index is already true of the native Store API's own product-list query (no catalog split exists anywhere in this platform's data model); the catalog badge itself only needs the same `food_details`/`wine_details` presence check Product Listing's card already uses for its own catalog-fact slot; sort, Load More, and empty states are Product Listing's own already-built infrastructure, reusable as-is. Building this now — rather than waiting for Meilisearch — directly serves the new business requirement without inventing anything Meilisearch-specific ahead of its approval.
- **Date:** 2026-07-19
- **Impact:** Implemented §5 (unified scope), §6 (URL-reflected state), §14 (sort, relabeled), §18/§19 (zero-result vs. pre-query states), §22 (result-count announcement), and §25 (`noindex, follow`) — see `storefront/README.md`'s "Milestone 10" section for full detail. Decisions worth recording explicitly:
  1. **The card's one-fact slot now has two possible occupants depending on context, never both at once.** On category/collection listings it shows Food Central's prep-time fact (Milestone 9); on search results (`showCatalogBadge`) it shows the catalog-identity badge instead. This is a `ProductPreview` prop threaded through the same shared `PaginatedProducts` component both surfaces already use, not a second card implementation — directly satisfying `04_PRODUCT_LISTING_SPECIFICATION.md` §9's "same card hierarchy... holds across search results" rule while still respecting its one-slot-per-card cap.
  2. **Autocomplete (§7) was considered and deliberately not built**, distinct from the other Meilisearch-gated mechanisms (which were simply out of scope): a native, non-typo-tolerant "as you type" suggestion list would likely return nothing for the exact single-character-typo case §7/§8 exist to solve, and shipping a version that fails at its own stated purpose would be a worse outcome than not shipping it yet, per `EXPERIENCE_PRINCIPLES.md` #9 (Trust Must Be Visible). Recorded as a deliberate quality judgment, not an oversight.
  3. **Zero-result recovery is honestly two-tier, not three.** §18 names typo-correction, closest-category, and curated-fallback as the priority order. Typo-correction needs real typo-distance data this implementation doesn't have; building a fake "closest match" (e.g., a naive substring heuristic) risks presenting a guess as if it were a genuine recovery suggestion, which §18 itself treats as a trust question, not just a UX one. The zero-result state here is a plain "no results" message plus a browse-back link only — an honest subset of the specified behavior, not a compromised version of it.
  4. **Search-within-category (§4, §15/§16) was not built.** The entry-point mechanics `01_NAVIGATION_SPECIFICATION.md` §15 already establish are unchanged; what's missing is a scoped-search affordance on category pages themselves. Not attempted this milestone — recorded as a real gap, not resolved.
  5. **Temporary QA seed data, created and fully removed within this milestone — with a cleaner cleanup than Milestone 9's.** Three products (one Food Central, two Wine & Spirits, all sharing the word "Citrus" in their titles so a single real query could prove the cross-catalog badge behavior against genuine data, not by construction) were created via a fresh `claude-qa2@liquorcentral.ng` admin account and deleted immediately after screenshots. Both this milestone's QA account and Milestone 9's still-open `claude-qa@liquorcentral.ng` leftover were then cross-deleted from each other (Medusa allows a user to delete a *different* user even though it cannot delete itself) — resolving Milestone 9's own open cleanup item as a side effect, not just avoiding a new one. A harmless orphaned auth-identity credential remains for each deleted account (the login credential itself isn't cascade-deleted with the User record) — confirmed non-functional: authenticating with it still issues a JWT, but every admin endpoint rejects it as Unauthorized, since no User record backs it. Not a security concern, noted for completeness.
  Real axe-core testing against the pre-query, populated-results, and zero-result search states with real QA data present found zero new violations — only the same pre-existing, already-documented contrast issues (Milestones 7–9). A structural check confirmed the `noindex, follow` robots directive, zero nested `<a>` elements, correct catalog-badge text on all three seeded products, and that selecting a sort order updates the URL to carry both `q` and `sortBy` together, with the result grid re-sorting correctly.
  `docs/PROJECT_STATUS.md` (→ v5.6), `docs/AI_HANDOFF.md` (→ v5.2), and `docs/CHANGELOG.md` (new entry) updated to reflect completion; `storefront/README.md` gained a full "Milestone 10" section.
- **Status:** Final for Milestone 10. No Frozen specification was edited. Meilisearch-backed typo tolerance, synonyms, autocomplete, editorial boosting, faceted search results, and search-within-category were all deliberately not built, each named as a real, tracked gap rather than silently skipped. The next unblocked step, per the confirmed specification order, is Product Details (`05_PRODUCT_DETAILS_SPECIFICATION.md`).

### Engineering Milestone 11: Product Details (`05_PRODUCT_DETAILS_SPECIFICATION.md`) — three genuine pre-existing bugs found via click-through testing; first use of a real stock location

- **Decision:** Built Product Details next per the confirmed specification implementation order — Cart's own turn (`06_CART_SPECIFICATION.md`) follows. Rebuilt the vendored starter's generic product detail page around the specification's real information hierarchy, fact sheets, and trust/delivery information, rather than a from-scratch redesign of its existing 3-column layout.
- **Reasoning:** The vendored PDP was T-shirt-store-shaped placeholder content that had never been touched since the Milestone 5 scaffold — "Material/Country of origin/Weight," "Fast delivery in 3-5 business days," "Easy returns, no questions asked." None of it is true of this business (Food Central is same-day, not multi-day; the alcohol return policy is a genuinely open decision, not "no questions asked" for either catalog), so leaving it in place would mean a customer-facing page actively stating false information — a direct trust violation, not a cosmetic gap. Reusing the existing layout structure (rather than rebuilding it) was a deliberate scope decision: the 3-column arrangement already roughly satisfies §5's "buy section above the fold, facts below" ordering, and rewriting it would have been effort spent on something not actually broken.
- **Date:** 2026-07-19
- **Impact:** Implemented §7 (Information Hierarchy — an above-the-fold key-facts line, distinct from the full fact sheet), §10–§13 (wine/food fact sheets, reading the same `wine_details`/`food_details` fields Product Listing's card already uses), §17 (a real quantity stepper — previously entirely absent), §18/§25 (add-to-cart confirmation), §19–§21 (honest, catalog-specific trust/delivery information), §27 (real per-product SEO metadata and `Product`/`Offer` structured data). Findings worth recording explicitly:
  1. **Three genuine bugs found via real click-through testing, none caught by `tsc`, ESLint, Jest, or axe-core.** (a) A button-copy bug: `!selectedVariant && !options` was meant to detect "nothing chosen yet," but `options` starts as `{}`, and an empty object is truthy in JavaScript — `!options` was always `false`, so the "Select variant" branch was unreachable code; a multi-variant product with nothing selected showed "Out of stock" instead, a direct §8 "never misleading" violation (a real "From ₦24,500" price sitting next to a false "Out of stock" claim). (b) An accessibility defect in the shared `Accordion` component: the visible section title sat outside `AccordionPrimitive.Trigger` entirely, so only a small chevron icon (under the 44×44px touch-target minimum) was actually clickable — a genuine §25 violation caught only by attempting to click the title itself, the control's obvious natural target, and observing nothing happened. (c) A pre-existing crash bug present since the Milestone 5 scaffold, never previously exercised by any check or test: `getImagesForVariant(pricedProduct, ...)` was called *before* the `if (!pricedProduct) notFound()` check, so a genuinely nonexistent product handle threw an unhandled `TypeError` and rendered the generic error boundary instead of the graceful not-found page — a direct §24 violation ("never a raw, dead-end error"). All three fixed and re-verified with real screenshots before/after.
  2. **A same-catalog gap in Related Products, found and fixed.** A product with no category, collection, or tags at all fell through to an entirely unfiltered "all products" query, meaning a Food Central dish's "You might also like" could surface Wine & Spirits results and vice versa — never §15's intent. Fixed by filtering to the same catalog (`food_details`/`wine_details` presence, the same mechanism already established for Product Listing's card and Search's badge) when no category exists to narrow by.
  3. **The alcohol return/refund policy remains genuinely undecided, and the Wine & Spirits PDP now visibly reflects that** — no returns claim renders at all for that catalog, per §19's own explicit instruction that a PDP "states only what is actually decided," rather than a placeholder. This is a direct, visible consequence of an open business decision, not an oversight; Food Central's returns statement ("not eligible for return, cooked to order") is a factual consequence of the business model, not a policy invention.
  4. **First real stock location and inventory levels created in this Medusa instance.** No stock location existed anywhere before this milestone (confirmed via the Admin API) — meaning every "in stock" product built so far (including Product Listing's and Search's own QA products) relied entirely on `manage_inventory: false` (uncapped, always-purchasable) rather than genuine tracked stock. Demonstrating §17's actual stock-cap requirement needed a real location and real inventory levels, created temporarily via the Admin API and removed along with the QA products afterward. This is infrastructure plumbing (enabling inventory tracking to function at all), not a business decision — no location name, address, or operational detail here should be read as a real decision about warehouse location; a real one is Paul's call whenever fulfillment planning reaches that point.
  5. **Gallery zoom (§6) is a genuine, tracked gap, not an oversight.** Explicitly required by the specification; the vendored gallery has no zoom mechanism of any kind. A real interactive feature (state management, keyboard operability, touch-gesture support) deliberately deferred to its own focused pass rather than rushed alongside everything else in this milestone — the same discipline Search applied to deferring autocomplete (Milestone 10).
  Real axe-core testing against the wine PDP, food PDP, sold-out PDP, and the (now-fixed) not-found page, with real QA data present, found zero new violations on three of the four pages; the fourth shows only the same pre-existing, already-documented `text-ui-fg-muted` price-text contrast issue on a related-product card (Milestones 7–9). A structural check confirmed valid JSON-LD, a real unique page title, zero nested `<a>` elements, and the accordion trigger's `aria-expanded` correctly flipping on a keyboard `Enter` press.
  Both temporary admin accounts used for this milestone's QA seeding were cross-deleted cleanly from each other, leaving no live leftover account — Milestone 9's own still-open cleanup item (a temporary account that cannot delete itself) remains isolated to that milestone, not recurring.
  `docs/PROJECT_STATUS.md` (→ v5.7), `docs/AI_HANDOFF.md` (→ v5.3), `docs/ROADMAP.md` (→ v5.9), and `docs/CHANGELOG.md` (new entry) updated to reflect completion; `storefront/README.md` gained a full "Milestone 11" section including the visual-validation screenshots' findings.
- **Status:** Final for Milestone 11. No Frozen specification was edited. Gallery zoom, Pairing Recommendations (§14, blocked on the unbuilt "pairs with" data model), and Gift Wrap cross-sell (§16, needs a new backend mechanism the specification's own Backend Requirements table already marks not yet built) were all deliberately not built. The payment provider, notification provider, and Food Central/Wine & Spirits fulfillment tracks remain exactly as blocked as Milestone 10 left them, except Checkout specifically, which is now unblocked (Paystack approved). ### Batch of business decisions approved by Paul — payment provider, notifications, delivery areas/pricing, operating hours, inventory policy, search presentation, age verification, checkout, currency, tax, coupons, loyalty, wishlist; standing "continue building" authorization; specification implementation order confirmed

- **Decision:** Paul supplied a large batch of previously-open business decisions in a single direction, all approved unless he explicitly revisits one later:
  - **Payments:** local payment provider is **Paystack**. Build the integration now, via environment variables, without waiting for real API keys (test credentials to follow). **Cash-on-delivery is not offered.**
  - **Notifications:** three channels — **Email, WhatsApp, and in-app** — with Email and WhatsApp both mandatory. The notification system must be built so channels can be enabled/disabled/added later without an architectural rewrite.
  - **Delivery areas:** Food Central delivers **within Lagos Island** at launch; Wine & Spirits delivers **across all of Lagos** at launch (narrower than `BUSINESS_RULES.md`'s existing "nationwide" framing for Wine & Spirits — see Impact, below). Both areas must be configurable later without a code change.
  - **Delivery pricing:** distance-based, starting around ₦3,000, fully configurable — never hardcoded. Exact pricing rules remain a later, separate decision.
  - **Operating hours:** 9:00 AM–11:00 PM initially, configurable later without a code change.
  - **Inventory:** zero stock displays "Out of Stock"; no backorders.
  - **Search:** one unified result list across both catalogs (never split into separate result pages), with a small visual badge/tag per result indicating Food Central vs. Wine & Spirits.
  - **Age verification (MVP):** a required checkbox — "I confirm that I am 18 years or older" — replacing any stronger mechanism (DOB entry, ID verification, third-party verification) for now. The architecture must allow swapping in a stronger mechanism later without a major refactor.
  - **Checkout:** guest checkout remains allowed; a guest must still supply everything needed to fulfil the order; account creation is encouraged (not forced) immediately after a successful guest checkout, pre-filled from the order just placed.
  - **Currency:** NGN only at launch (already the project's existing assumption — reaffirmed, not new).
  - **Tax:** VAT is displayed during checkout; pricing is not assumed tax-inclusive unless an Approved document explicitly says so (none currently does).
  - **Coupons:** build the capability (a redeemable-code discount mechanism); do not create any actual coupon yet.
  - **Loyalty:** confirmed out of scope for MVP (reaffirms what every specification's own Future Expansion section already said).
  - **Wishlist:** remains long-term roadmap only; not to be prioritized ahead of launch-critical commerce work unless it naturally falls out of a current milestone.
  - **Process:** from this point forward the default is to keep building milestone by milestone without stopping for approval, interrupting only for a genuine business decision, a genuine architectural conflict, or a contradiction with an Approved document — explicitly extending the autonomous-continuation authorization first given after Milestone 7 into a durable standing rule, not a one-time grant. Missing credentials/API keys are never a reason to stop: implement against environment variables, leave placeholders, validate everything validatable without live credentials, and document what couldn't be validated. The specification implementation order is confirmed as Navigation → Homepage → Search → Product Listing → Product Details → Cart → Checkout → Customer Account → Food Ordering → Delivery Tracking, "unless objective architectural evidence suggests otherwise."
- **Reasoning:** These were the load-bearing open items `IMPLEMENTATION_READINESS_REPORT.md` §3 and `PROJECT_STATUS.md`'s "Decisions awaiting Paul's approval" had been carrying since planning concluded — most urgently the payment-provider choice, the project's sole confirmed launch blocker. Supplying them now unblocks Checkout, real order communication, and Wine & Spirits' nationwide-delivery-mechanism track, none of which engineering could resolve on its own.
- **Date:** 2026-07-19
- **Impact:**
  1. **Payment provider is no longer a blocker.** `ROADMAP.md` Phase 1 and `07_CHECKOUT_SPECIFICATION.md` §14/§26's Payment State Behaviour can now be implemented against Paystack specifically, using environment variables for the not-yet-supplied API keys — no engineering work waits on this any further.
  2. **A genuine tension surfaced, not silently resolved:** `BUSINESS_RULES.md` and `docs/AI_HANDOFF.md` §3 both currently state "wine and spirits ship nationwide across Nigeria" as a finalized business rule. This decision scopes Wine & Spirits delivery to **all of Lagos at launch**, which reads as a launch-scope narrowing of that rule (a phased rollout), not a reversal of the long-term nationwide intent — the two are not contradictory once read as "nationwide is the eventual target; Lagos-wide is the launch configuration," but `BUSINESS_RULES.md` itself is Frozen and was not edited here, since this is an implementation-scope decision, not a request to revise that document's content. This decision does not authorize standing up Meilisearch; it authorizes building the unified-list-with-badges *presentation* against whatever search mechanism is live at the time, native today, Meilisearch later without a presentation-layer rewrite.

### Engineering Milestone 9: Product Listing (`04_PRODUCT_LISTING_SPECIFICATION.md`); temporary QA-only seed data for visual validation

- **Decision:** With Homepage (Milestone 8) complete, and per Paul's standing "continue the project" authorization (Engineering Milestone 7's entry, below), Product Listing was chosen as the next milestone — unblocked by any external dependency (unlike Search's Meilisearch gate or Checkout's payment-provider gate), and the natural next step since Navigation's mega menu and Homepage's Food Central Spotlight both already link into listing pages. Separately: because this project has never seeded a single catalog product, and the Product Card Information Hierarchy this milestone implements is meaningless to visually validate against an empty grid, 13 temporary `[QA]`-prefixed products were created directly via the Admin API for screenshot purposes and deleted immediately afterward.
- **Reasoning:** Product Listing's card is, per its own specification, "the single most-repeated component on the platform" — validating it against a genuinely empty catalog would satisfy the letter of the standing visual-validation requirement while missing its entire point (every card-specific state — the one-fact slot, both quick-add weights, "Select options," "Sold out" — literally cannot render with zero products). Fabricating screenshots (mockups, generated images) is explicitly forbidden by the standing requirement; the alternative chosen here is neither that nor a permanent change to the "zero catalog seeded" project state, which remains a deliberate decision (the wine/food attribute field lists are still open, per `docs/PROJECT_STATUS.md`) that this milestone had no basis to override. ### Two focused product-attribute modules, not one universal module

- **Decision:** Wine attributes and food attributes are modeled as two separate custom modules (`wine-details`, `food-details`), each linked 1:1 to the native Product module, rather than one shared module covering both.
- **Reasoning:** A wine has no spice level; a dish has no vintage. One shared module would accumulate irrelevant nullable fields per product and complicate validation.
- **Date:** 2026-07-18
- **Impact:** `PRODUCT_CATALOG.md`, `MEDUSA_EXTENSIONS.md` #1 and #2.
- **Status:** Approved as an architecture pattern; exact field lists per module still open.

### Storefront: Next.js Starter recommended over a from-scratch build

- **Decision:** Build on Medusa's official Next.js Starter ("DTC Starter") rather than a fully custom React application.
- **Reasoning:** The starter already has cart, checkout, region/tax handling, and payment integration wired and tested against Medusa's API; a from-scratch build would re-implement the same logic with no maintained baseline to diff against on upgrade. Its App Router/SSR foundation also directly supports organic product-page discovery.
- **Date:** 2026-07-18
- **Impact:** `TECH_STACK.md`.
- **Status:** Recommended — not yet formally approved by Paul.

### Search: Meilisearch recommended over Algolia, Elasticsearch, and Typesense

- **Decision:** Use Meilisearch as the search/facet engine.
- **Reasoning:** The only option that is both open-source (cost and data control) and officially documented for Medusa integration (lowest integration risk); its faceting fits the wine/food attribute filtering this project needs. Elasticsearch and Typesense have no official Medusa integration guide; Algolia is hosted-only with usage-based pricing that scales unfavorably at marketplace-adjacent scale.
- **Date:** 2026-07-18
- **Impact:** `MEDUSA_EXTENSIONS.md` #6, `TECH_STACK.md`, `ROADMAP.md` Phase 6.
- **Status:** Recommended — not yet formally approved by Paul.

### CMS: Sanity recommended over Contentful, Strapi, and Payload

- **Decision:** Use Sanity for editorial/marketing content, kept strictly separate from commerce data (which stays in Medusa), synced one-way (Medusa → Sanity only).
- **Reasoning:** Real-time collaborative editing, strong localization support, structured content that pairs well with the Next.js storefront, and an official Medusa integration guide built around exactly this use case.
- **Date:** 2026-07-18
- **Impact:** `MEDUSA_EXTENSIONS.md` #7, `PRODUCT_BLUEPRINT.md` §12.
- **Status:** Recommended, low urgency — not yet formally approved.

### Authentication: Medusa Auth confirmed as the required system of record

- **Decision:** All authentication (customer and admin) uses Medusa's native auth system. Third-party identity providers (e.g. Clerk, Auth.js) are not used for v1, and would only ever act as a bridge into Medusa Auth if ever needed, never a replacement.
- **Reasoning:** Every protected Medusa route and its RBAC system are built directly on Medusa's own identity model — there is no way to substitute it wholesale without breaking the rest of the framework.
- **Date:** 2026-07-18
- **Impact:** `TECH_STACK.md`.
- **Status:** Final (architectural constraint, not a preference).
